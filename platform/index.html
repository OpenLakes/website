Architecture

The OpenLakes platform in one glance

Eight documented layers mirror the structure of the OpenLakes Core repository. Each Helm chart is versioned, reviewed, and wired for ingress, storage, and metrics so you can inspect every manifest before applying it to your cluster.

Layer 01 · Infrastructure

Traefik ingress, Longhorn, PostgreSQL, MinIO, Kafka, and Nessie routing form the base.

Layer 02 · Compute

Trino 478 and Spark 4.1.0-preview3 (Iceberg + Nessie enabled) share object storage and credentials.

Layer 03 · Streaming

Reserved for future engines. Spark Structured Streaming ships today for event use cases.

Layer 04 · Orchestration

Airflow runtime image with OpenLakes providers and Papermill support keeps jobs reproducible.

Layer 05 · Analytics

Superset 4.1 and JupyterHub 4.3 connect to the same catalogs, buckets, and SSO settings.

Layer 06 · Ingestion

Meltano 4.0.6 projects and Debezium 3.0 Kafka Connect deliver repeatable ELT + CDC paths.

Layer 07 · Catalog

OpenMetadata 1.10.7 ships with PostgreSQL + OpenSearch backing services already plumbed.

Layer 08 · Monitoring

kube-prometheus-stack bundles Prometheus, Alertmanager, Grafana, and Loki dashboards.

Resource profiles

The deploy script inspects your cluster and picks a profile (compact-vm, compact, small, or full). Override it in core-config.yaml when you need a specific footprint.

compact-vm

  • Auto-selected for single-node macOS/VM setups.
  • Guards against under-provisioned CPU and memory.
  • Keeps every layer online with the smallest requests possible.

compact

  • Manual override for single-node bare metal.
  • Reduces Spark, Trino, Meltano, Debezium, and Loki footprints.
  • Useful when you need the entire stack on a workstation.

small

  • Default for most lab clusters and small multi-node installs.
  • Balances resource requests so all services remain enabled.
  • Works up to roughly 24 cores / 128 Gi across the cluster.

full

  • Selected when the cluster advertises ample CPU and memory.
  • Keeps higher retention, multiple Spark executors, and beefier Trino workers.
  • Designed for production environments with dedicated hardware.
