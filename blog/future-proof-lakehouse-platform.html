Build a Future-Proof Data Platform Without Vendor Lock-In

Why Apache Iceberg and open source components protect your long-term technology strategy.

Every technology leader faces a fundamental tension: adopt managed services for speed, or build on open standards for control. With data infrastructure, this choice has long-term strategic implications. Proprietary table formats trap your data. Managed platforms create switching costs. OpenLakes offers an alternative.

The Vendor Lock-In Problem

Proprietary data platforms create lock-in through:

  • Proprietary table formats — Snowflake tables and Databricks Delta Lake require their engines to read efficiently
  • Platform-specific SQL extensions — Queries written for one platform don't port cleanly to others
  • Integrated tooling — BI, governance, and ingestion tools designed for vendor ecosystems
  • Data gravity — Moving petabytes out of a cloud region is expensive and slow

Once locked in, switching costs become prohibitive. You're subject to annual price increases with limited negotiating power.

OpenLakes: Open Standards by Design

OpenLakes builds on Apache Iceberg, the open table format created by Netflix and now governed by the Apache Software Foundation.[1] Every component in the stack is open source and interoperable.

Apache Iceberg tables are engine-agnostic. The same tables can be read and written by Spark, Trino, Flink, Dremio, Snowflake, and dozens of other engines. Your data is never trapped.

Architecture Overview

OpenLakes implements a layered architecture where each component is replaceable:[2]

Layer 08: Monitoring → Prometheus, Grafana, Loki (replaceable with Datadog, etc.) Layer 07: Catalog → OpenMetadata (replaceable with DataHub, Atlan) Layer 06: Ingestion → Meltano, Debezium (replaceable with Airbyte, Fivetran) Layer 05: Analytics → Superset, JupyterHub (replaceable with Metabase, Looker) Layer 04: Orchestration → Airflow (replaceable with Dagster, Prefect) Layer 02: Compute → Spark, Trino (add Flink, Dremio, StarRocks) Layer 01: Infrastructure → MinIO, PostgreSQL, Kafka, Nessie

The Nessie Catalog

OpenLakes uses Project Nessie as its Iceberg REST catalog.[3] Nessie provides:

  • Git-like versioning — Branch, tag, and merge table changes
  • Cross-engine compatibility — Standard REST API for any Iceberg-compatible engine
  • ACID transactions — Consistent reads and writes across concurrent queries
  • Time travel — Query historical snapshots of your data

The catalog metadata is stored in PostgreSQL,[4] not a proprietary service. You can export, migrate, or replicate it at will.

Multi-Engine Query Architecture

OpenLakes deploys both Apache Spark and Trino as query engines:[5]

Engine Version Best For
Apache Spark Custom image with Iceberg/S3A[6] ETL, ML pipelines, streaming, large batch jobs
Trino 478[7] Interactive queries, BI dashboards, ad-hoc analysis

Both engines read from the same Iceberg tables through the Nessie catalog. Choose the right engine for each workload, or add additional engines as needs evolve.

Storage Independence

Data is stored in MinIO using S3-compatible APIs.[8] The storage layer supports:

  • Hot tier — NVMe/SSD storage mapped from host paths for high performance
  • Cold tier — Optional NAS/NFS mounts for lifecycle tiering
  • Cache tier — Optional Alluxio integration for distributed caching[9]

Storage tiers are configured in a single YAML file.[10] Data is stored in open Parquet format within Iceberg tables—readable by any tool that understands Parquet.

Future-Proofing Your Architecture

With OpenLakes, you retain full optionality:

  • Add engines — Connect Flink, Dremio, or StarRocks to the same Iceberg tables
  • Change clouds — Deploy on AWS, GCP, Azure, or on-premises
  • Swap components — Replace Superset with Metabase, Meltano with Airbyte
  • Export data — Standard formats mean zero conversion on migration
  • Hybrid deployments — Run some workloads in cloud, others on-premises
