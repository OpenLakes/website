Data Sovereignty: Keep Your Data in Your Environment

How self-hosted data infrastructure addresses compliance, security, and privacy requirements.

For teams operating in regulated industries, cloud data warehouses present a fundamental challenge: your most sensitive data lives in someone else's environment. OpenLakes keeps the entire stack on infrastructure you control.

Important: OpenLakes is open-source infrastructure software. Security implementation is your responsibility. OpenLakes does not guarantee data privacy, security, or protection against data loss. See our Terms of Service for details.

The Data Sovereignty Challenge

Regulated industries face strict requirements around data handling:

  • HIPAA — Healthcare data must be protected with specific safeguards
  • GDPR — European personal data may require EU-based processing
  • SOC 2 — Service organization controls for data security
  • PCI-DSS — Payment card data requires specific protections
  • ITAR/EAR — Defense and export-controlled data restrictions

Cloud data warehouses complicate compliance. Your data transits third-party networks, lives in multi-tenant environments, and may be subject to foreign jurisdiction requests.

Self-Hosted Data Infrastructure

OpenLakes deploys entirely within your Kubernetes cluster.[1] Data flows through components you control:

No data egress. All processing happens within your environment. Data never transits external networks for analytics, transformation, or storage.

Network Architecture

OpenLakes uses Traefik as an ingress controller with configurable TLS termination:[2]

  • Internal service mesh — Components communicate via ClusterIP services
  • TLS termination — Let's Encrypt certificates via DNS-01 challenge, or bring your own
  • Network isolation — All services run in the openlakes namespace[3]
  • No external dependencies — No calls to external APIs or telemetry services

Storage Architecture

Data is stored in MinIO using S3-compatible APIs, with multiple storage tiers:[4]

Tier Purpose Control
Hot (MinIO) Active data, frequent access NVMe/SSD paths you specify[5]
Cold (NAS) Archive data, lifecycle tiering Your NFS/NAS mounts
Cache (Alluxio) Distributed caching layer Local SSD paths
Longhorn Replicated block storage Kubernetes PVCs with 2x replication[6]

All storage paths are configured in a single file you control:[7]

Authentication and Access Control

OpenLakes components support various authentication mechanisms:

  • PostgreSQL — Username/password authentication, configurable per database[8]
  • MinIO — Access key/secret key authentication[9]
  • Superset — Built-in user management[10]
  • JupyterHub — Configurable authenticator (dummy auth in dev, LDAP/OAuth in production)[11]
  • Airflow — Built-in RBAC[12]
  • OpenMetadata — Basic auth with JWT tokens[13]

Credentials are configured in core-config.yaml before deployment. For production, integrate with your identity provider via each component's native SSO support.

Audit and Observability

OpenLakes includes a complete observability stack for audit and monitoring:[14]

  • Prometheus — Metrics collection with 15-day retention[15]
  • Grafana — Dashboards for all components
  • Loki — Centralized log aggregation with 7-day retention[16]
  • Alertmanager — Configurable alerting rules

Query logs, access patterns, and system events are captured locally. Integrate with your SIEM by shipping logs from Loki.

Data Governance with OpenMetadata

OpenMetadata provides data cataloging and governance:[17]

  • Data discovery — Catalog all tables, columns, and schemas
  • Lineage tracking — Understand data flow through pipelines
  • Classification — Tag sensitive columns (PII, PHI, financial)
  • Ownership — Assign data stewards and owners
  • Quality rules — Define and monitor data quality expectations

Security Considerations

While OpenLakes provides the infrastructure, security implementation requires additional work:

Your Responsibilities

  • Network security — Firewall rules, network policies, VPN access
  • Encryption at rest — Configure Kubernetes secrets encryption, disk encryption
  • Encryption in transit — Ensure TLS is configured for all endpoints
  • Access management — Integrate with your identity provider
  • Patch management — Keep components updated
  • Backup and recovery — Implement your own backup strategy
  • Vulnerability scanning — Scan container images and dependencies

What OpenLakes Provides

  • No external telemetry — Components don't phone home
  • Configurable credentials — All passwords specified at deployment time
  • Namespace isolation — All components in a single namespace
  • Longhorn replication — Data replicated across nodes for availability[6]
  • Open source transparency — Full source code available for audit

Compliance Framework Mapping

Requirement Cloud DW Challenge OpenLakes Approach
Data residency Limited region options Deploy in any jurisdiction
Third-party access Vendor has access You control all access
Audit trails Vendor-provided logs Self-hosted logging stack
Encryption keys Often vendor-managed Your keys, your control
Network isolation Shared infrastructure Your network boundaries

What to evaluate before self-hosting

OpenLakes may be appropriate when:

  1. Data cannot leave your environment due to regulatory or policy requirements
  2. You need full audit control over data access and processing
  3. Third-party data processing creates unacceptable risk
  4. You have the security expertise to operate self-hosted infrastructure
  5. Compliance frameworks are easier to satisfy with on-premises control
